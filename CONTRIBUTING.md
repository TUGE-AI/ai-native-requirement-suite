# Contributing

1. Open an issue first for bugs, feature requests, or scope changes.
2. Keep public issues and PRs free of internal company names, customer names, private paths, and secret material.
3. Use one branch per change.
4. Link the PR back to the issue.
5. Include verification steps in the PR description.

## Suggested Flow

```text
issue
  -> branch
  -> changes
  -> local verification
  -> commit
  -> push
  -> pull request
  -> review
  -> merge
```

## Public vs Private

- Public repo: reusable content, public issues, and public-safe examples.
- Private repo: internal requirements, sensitive samples, and company-specific validation.
- Local workspace: temporary analysis only.
